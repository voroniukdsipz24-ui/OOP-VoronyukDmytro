
# Порушення принципу LSP (Liskov Substitution Principle)

## Приклади порушення LSP

### 1. **Клас Круг і клас Еліпс**

#### Порушення LSP:
Уявімо, що клас **Круг** є підкласом класу **Еліпс**. У класі **Еліпс** ми маємо два параметри: **ширина** і **висота**, які можуть бути різними, але у класі **Круг** ці значення мають бути рівними. Це обмежує можливості використання **Круга** як підкласу **Еліпса**.

#### Проблеми:
- **Обмежена поведінка**: При заміни **Еліпса** на **Круг** програма може почати поводитись некоректно через жорстке обмеження рівності параметрів.

#### Рішення:
Рекомендується розділити класи **Еліпс** і **Круг** на окремі класи, а не наслідувати **Круг** від **Еліпса**. Це дозволить уникнути обмежень і зберегти гнучкість програми.

### 2. **Клас Трикутник і клас Рівносторонній Трикутник**

#### Порушення LSP:
Клас **Рівносторонній Трикутник** є підкласом **Трикутника**, але він має додаткову умову, що всі сторони повинні бути однаковими. Це порушує принцип LSP, оскільки якщо ми замінимо об'єкт **Трикутник** на **Рівносторонній Трикутник**, поведінка програми може змінитися через необхідність підтримки рівних сторін.

#### Проблеми:
- **Невідповідність інтерфейсу**: Програма, яка працює з **Трикутником**, не може коректно працювати з **Рівностороннім Трикутником** без змін.

#### Рішення:
Рекомендується реалізувати окремі класи для **Трикутника** і **Рівностороннього Трикутника**, щоб забезпечити коректну роботу кожного з них відповідно до своїх обмежень.

### 3. **Клас Автомобіль і клас Електричний Автомобіль**

#### Порушення LSP:
Клас **Автомобіль** має методи для заправки, але **Електричний Автомобіль** потребує методів для зарядки. Якщо замінити **Автомобіль** на **Електричний Автомобіль**, програма може почати працювати некоректно через відсутність підтримки зарядки в методах базового класу.

#### Проблеми:
- **Невідповідність методів**: Програма, що працює з **Автомобілем**, не може коректно працювати з **Електричним Автомобілем**, якщо вона не враховує специфіку зарядки.

#### Рішення:
Можна створити спільний інтерфейс або абстрактний клас для **Автомобіля**, в якому будуть оголошені загальні методи (наприклад, для обслуговування), а вже конкретні класи **Автомобіль** і **Електричний Автомобіль** реалізують специфічні методи для заправки та зарядки.

## Висновки

Принцип Liskov Substitution Principle (LSP) є одним з основних принципів об'єктно-орієнтованого програмування. Він гарантує, що підкласи можуть бути замінені базовими класами без порушення коректності програми. Порушення цього принципу може призвести до некоректної роботи програми, зниження гнучкості та утруднень у підтримці коду. Для забезпечення дотримання LSP слід ретельно проектувати класи і використовувати інтерфейси або абстрактні класи, що дозволяють забезпечити однакову поведінку для різних типів об'єктів.

## Команди для запуску тестів

1. Для перевірки коректності реалізації класів, необхідно виконати модульні тести, що перевіряють заміну підкласів без змін поведінки програми.
2. Запуск тестів здійснюється за допомогою стандартного тестового фреймворку:

```bash
python -m unittest test_lsp.py
```

Тести повинні перевірити правильність функціонування заміни підкласів у різних умовах.
