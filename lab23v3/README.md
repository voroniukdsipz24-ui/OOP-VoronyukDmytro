# Лабораторна робота №23: Реалізація принципів ISP та DIP через Dependency Injection

## Опис

Цей проєкт демонструє застосування принципів **Розділення Інтерфейсу (ISP)** та **Інверсії Залежностей (DIP)** для рефакторингу існуючого коду. Ми реалізуємо **Dependency Injection (DI)** через конструктор для зменшення зв'язаності між класами та покращення тестованості. Проєкт зосереджений на обробці платіжних операцій через різні платіжні шлюзи: картки, PayPal та криптовалюту.

## Мета

- **ISP (Interface Segregation Principle)**: Кожен клієнт повинен залежати тільки від методів, які він використовує. Ми виділяємо інтерфейси для кожного платіжного шлюзу, що дозволяє клієнтам залежати лише від тих методів, які їм необхідні.
- **DIP (Dependency Inversion Principle)**: Модулі вищого рівня не повинні залежати від модулів нижчого рівня. У нашому випадку, ми застосовуємо **Dependency Injection** для того, щоб платіжні шлюзи (нижчий рівень) не створювалися безпосередньо в класі `PaymentHandler`.

## Реалізація принципів

### 1. Розділення Інтерфейсу (ISP)

Принцип ISP передбачає, що клієнти повинні залежати лише від тих методів, які вони використовують. У нашому випадку, ми створили інтерфейс `IPaymentGateway`, який реалізують усі платіжні шлюзи (картка, PayPal, криптовалюта). Кожен клієнт отримує доступ лише до необхідних методів.

### 2. Інверсія Залежностей (DIP)

Модулі вищого рівня (наприклад, `PaymentHandler`) не повинні залежати від конкретних реалізацій нижчого рівня (наприклад, `CreditCardGateway`, `PayPalService`). Замість цього, ми використовуємо **Dependency Injection**, що дозволяє класам отримувати залежності через конструктор і легко змінювати реалізації без зміни основного коду.

### 3. Dependency Injection

Ми використовуємо Dependency Injection через конструктор у класі `PaymentHandler`, що дозволяє нам передавати будь-який платіжний шлюз (картку, PayPal чи криптовалюту) без потреби змінювати код самого класу.

## Опис змін

1. Створення інтерфейсу `IPaymentGateway`, який реалізується всіма платіжними шлюзами.
2. Розподілення логіки на окремі класи: `CreditCardGateway`, `PayPalService` та `CryptoExchange`.
3. Використання Dependency Injection для впровадження залежностей через конструктор класу `PaymentHandler`.



## Висновки

1. Реалізація принципів **ISP** та **DIP** дозволяє зробити систему гнучкішою та зменшити зв'язність між класами.
2. **Dependency Injection** через конструктор дає можливість легко змінювати реалізації залежностей, що покращує тестування та розширюваність.
3. Рішення допомагає уникнути проблем з масштабуванням, оскільки нові платіжні шлюзи можуть бути додані без змін у існуючому коді.
